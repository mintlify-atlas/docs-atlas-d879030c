title: Secrets Management
description: Securely manage passwords, API keys, and sensitive configuration

Kamal provides a secure system for managing sensitive configuration like passwords, API keys, and tokens through its secrets management system.

## How Secrets Work

Secrets in Kamal are:
- Stored in `.kamal/secrets` files (gitignored by default)
- Loaded as environment variables during deployment
- Never logged or displayed in command output
- Can be fetched from external vaults

<Warning>
Never commit `.kamal/secrets` files to version control. Add them to `.gitignore`.
</Warning>

## Secrets Files

### Default Secrets Location

Kamal looks for secrets in:

```
.kamal/secrets         # Common secrets for all destinations
.kamal/secrets-common  # Alternative name
```

### Environment-Specific Secrets

For multiple environments:

```
.kamal/secrets-common     # Shared across all environments
.kamal/secrets.staging    # Staging-specific secrets
.kamal/secrets.production # Production-specific secrets
```

From `lib/kamal/secrets.rb:43-45`, Kamal loads secrets from both common and destination-specific files:

```ruby
def secrets_filenames
  [ "#{@secrets_path}-common", "#{@secrets_path}#{(.".#{@destination}" if @destination)}" ]
end
```

### Secrets File Format

Secrets use dotenv format:

```bash
# .kamal/secrets
KAMAL_REGISTRY_PASSWORD=ghp_xxxxxxxxxxxx
MYSQL_ROOT_PASSWORD=secure-database-password
REDIS_PASSWORD=secure-redis-password
RAILS_MASTER_KEY=abc123def456
SECRET_KEY_BASE=very-long-secret-key
```

## Using Secrets in Configuration

### Registry Authentication

```yaml
registry:
  server: ghcr.io
  username: myuser
  password:
    - KAMAL_REGISTRY_PASSWORD
```

In `.kamal/secrets`:
```bash
KAMAL_REGISTRY_PASSWORD=ghp_your_token_here
```

### Environment Variables

Pass secrets to your application:

```yaml
env:
  secret:
    - RAILS_MASTER_KEY
    - SECRET_KEY_BASE
    - DATABASE_PASSWORD
```

### Accessory Secrets

Secure your accessory services:

```yaml
accessories:
  db:
    image: mysql:8.0
    host: 192.168.0.2
    env:
      secret:
        - MYSQL_ROOT_PASSWORD
  
  redis:
    image: redis:7
    host: 192.168.0.3
    env:
      secret:
        - REDIS_PASSWORD
```

### Builder Secrets

Pass secrets during build time:

```yaml
builder:
  secrets:
    - GITHUB_TOKEN
    - NPM_TOKEN
```

Use in Dockerfile:
```dockerfile
RUN --mount=type=secret,id=GITHUB_TOKEN \
    GITHUB_TOKEN=$(cat /run/secrets/GITHUB_TOKEN) \
    bundle install
```

## Inline Command Substitution

Kamal supports command substitution in secrets files:

```bash
# .kamal/secrets
# Fetch from 1Password
DATABASE_PASSWORD=$(op read op://vault/item/password)

# Fetch from AWS Secrets Manager
AWS_SECRET=$(aws secretsmanager get-secret-value --secret-id my-secret --query SecretString --output text)

# Use environment variables
DEPLOY_KEY=${LOCAL_DEPLOY_KEY}

# Combine multiple sources
COMPLEX_SECRET=$(echo $LOCAL_VAR | base64)
```

From `lib/kamal/secrets.rb:4`, inline command substitution is enabled:

```ruby
Kamal::Secrets::Dotenv::InlineCommandSubstitution.install!
```

## Vault Adapters

### Using External Vaults

Kamal can fetch secrets from external vault services:

```bash
kamal secrets fetch SECRET_NAME --adapter 1password --account myaccount --from myvault
```

### 1Password Adapter

Fetch from 1Password:

```bash
# Fetch a single secret
kamal secrets fetch RAILS_MASTER_KEY \
  --adapter 1password \
  --account myaccount \
  --from myvault

# Use in secrets file
# .kamal/secrets
RAILS_MASTER_KEY=$(kamal secrets fetch RAILS_MASTER_KEY --adapter 1password --account myaccount --from myvault --inline)
```

Or directly with op CLI:

```bash
# .kamal/secrets
RAILS_MASTER_KEY=$(op read op://myvault/rails/master_key)
DATABASE_PASSWORD=$(op read op://myvault/database/password)
```

### AWS Secrets Manager

```bash
# .kamal/secrets
DATABASE_PASSWORD=$(aws secretsmanager get-secret-value --secret-id prod/db/password --query SecretString --output text)
API_KEY=$(aws secretsmanager get-secret-value --secret-id prod/api/key --query SecretString --output text)
```

### Azure Key Vault

```bash
# .kamal/secrets
DATABASE_PASSWORD=$(az keyvault secret show --name db-password --vault-name mykeyvault --query value -o tsv)
```

### Google Secret Manager

```bash
# .kamal/secrets
DATABASE_PASSWORD=$(gcloud secrets versions access latest --secret="db-password")
```

### HashiCorp Vault

```bash
# .kamal/secrets
DATABASE_PASSWORD=$(vault kv get -field=password secret/database)
API_KEY=$(vault kv get -field=key secret/api)
```

## Environment Variables

### Clear vs Secret

Distinguish between public and sensitive configuration:

```yaml
env:
  clear:  # Visible in container inspection
    RAILS_ENV: production
    RACK_ENV: production
    LOG_LEVEL: info
    DATABASE_HOST: 192.168.0.2
  
  secret:  # Hidden from logs and inspection
    - RAILS_MASTER_KEY
    - SECRET_KEY_BASE
    - DATABASE_PASSWORD
    - API_KEY
```

### Combined Configuration

Build connection strings from both:

```yaml
env:
  clear:
    DATABASE_HOST: 192.168.0.2
    DATABASE_PORT: 3306
    DATABASE_NAME: myapp_production
  secret:
    - DATABASE_PASSWORD
```

In your app:
```ruby
# config/database.yml
production:
  host: <%= ENV['DATABASE_HOST'] %>
  port: <%= ENV['DATABASE_PORT'] %>
  database: <%= ENV['DATABASE_NAME'] %>
  password: <%= ENV['DATABASE_PASSWORD'] %>
```

### Tags for Dynamic Values

Use tags to inject dynamic values:

```yaml
env:
  clear:
    VERSION: <%= ENV['VERSION'] || `git rev-parse HEAD`.strip %>
    DEPLOYED_AT: <%= Time.now.iso8601 %>
  secret:
    - SECRET_KEY_BASE
```

## Viewing Secrets

### Print Current Secrets

```bash
kamal secrets print
```

This shows all loaded secrets (be careful - displays sensitive data!).

### Check Secret Availability

From `lib/kamal/secrets.rb:30-34`, you can check if a secret exists:

```ruby
def key?(key)
  synchronized_fetch(key).present?
rescue KeyError
  false
end
```

### Debug Secret Loading

If secrets aren't loading:

```bash
# Check which files exist
ls -la .kamal/secrets*

# Verify file contents (be careful!)
cat .kamal/secrets

# Test with print command
kamal secrets print
```

## Best Practices

### File Organization

<Steps>

<Step title="Create .gitignore">
Ensure secrets are never committed:

```bash
# .gitignore
.kamal/secrets*
!.kamal/secrets.example
```
</Step>

<Step title="Create example file">
Document required secrets:

```bash
# .kamal/secrets.example
KAMAL_REGISTRY_PASSWORD=your-registry-token-here
MYSQL_ROOT_PASSWORD=secure-password-here
REDIS_PASSWORD=redis-password-here
RAILS_MASTER_KEY=32-character-hex-string
SECRET_KEY_BASE=long-random-string
```
</Step>

<Step title="Document in README">
Add setup instructions to your README:

```markdown
## Secrets Setup

1. Copy example secrets file:
   cp .kamal/secrets.example .kamal/secrets

2. Fill in actual values in `.kamal/secrets`

3. Never commit `.kamal/secrets` to version control
```
</Step>

</Steps>

### Rotation Strategy

<Accordion title="Regular rotation">
Rotate secrets periodically:

```bash
# 1. Generate new secrets
# 2. Update .kamal/secrets file
# 3. Deploy with new secrets
kamal env push

# 4. Reboot services
kamal app boot

# 5. Update accessories
kamal accessory reboot all
```
</Accordion>

<Accordion title="Emergency rotation">
If secrets are compromised:

1. Immediately generate new secrets
2. Update `.kamal/secrets`
3. Push and restart:
```bash
kamal env push
kamal deploy
kamal accessory reboot all
```
4. Revoke old secrets in external services
</Accordion>

### Access Control

<Accordion title="Limit secret access">
- Store secrets in encrypted vault (1Password, AWS Secrets Manager, etc.)
- Use IAM roles and policies for vault access
- Require authentication for secret fetching
- Audit secret access logs
</Accordion>

<Accordion title="Team access">
```bash
# Each team member fetches from shared vault
# .kamal/secrets (not committed)
RAILS_MASTER_KEY=$(op read op://shared-vault/rails/master_key)
DATABASE_PASSWORD=$(op read op://shared-vault/database/password)
```

Benefits:
- Secrets never in files on developer machines
- Centralized access control
- Audit trail of who accessed what
- Easy rotation without distributing new files
</Accordion>

## Security Considerations

### File Permissions

Secure your secrets files:

```bash
# Restrict access to secrets files
chmod 600 .kamal/secrets*

# Verify permissions
ls -la .kamal/secrets*
# Should show: -rw------- (600)
```

### Secure Transmission

Secrets are:
- Transmitted over SSH (encrypted)
- Stored in container environment (isolated)
- Not logged in Kamal output
- Not visible in `docker inspect` (when using secret env)

### Environment Isolation

Separate secrets by environment:

```yaml
# Production
servers:
  web:
    - prod-server-1

env:
  secret:
    - PROD_DATABASE_PASSWORD
    - PROD_API_KEY
```

```yaml
# Staging
servers:
  web:
    - staging-server-1

env:
  secret:
    - STAGING_DATABASE_PASSWORD
    - STAGING_API_KEY
```

## Common Patterns

### Database Connection URL

```bash
# .kamal/secrets
DATABASE_PASSWORD=secure-password
DATABASE_URL=postgresql://user:${DATABASE_PASSWORD}@192.168.0.2:5432/myapp_production
```

Or build in config:

```yaml
env:
  clear:
    DATABASE_HOST: 192.168.0.2
    DATABASE_USER: myapp
    DATABASE_NAME: myapp_production
  secret:
    - DATABASE_PASSWORD
```

### External Service Keys

```bash
# .kamal/secrets
AWS_ACCESS_KEY_ID=AKIA...
AWS_SECRET_ACCESS_KEY=...
STRIPE_API_KEY=sk_live_...
SENDGRID_API_KEY=SG...
S3_BUCKET=myapp-production
```

```bash
# .kamal/secrets
OAUTH_CLIENT_ID=...
```

## Troubleshooting

<Accordion title="Secret not found error">
Error message:
```
Secret 'SECRET_NAME' not found in .kamal/secrets
```

Solutions:
1. Check secret is defined in `.kamal/secrets`
2. Verify file name matches environment:
   ```bash
   ls -la .kamal/secrets*
   ```
3. Check for typos in secret name
4. Ensure file is not empty
</Accordion>

<Accordion title="Command substitution not working">
If `$(command)` isn't evaluated:

1. Ensure command is installed and in PATH
2. Test command separately:
   ```bash
   op read op://vault/item/field
   ```
3. Check for shell quoting issues
4. Verify command output is not empty
</Accordion>

<Accordion title="Secrets showing in logs">
If secrets appear in logs:

1. Use `secret:` section, not `clear:`:
   ```yaml
   env:
     secret:  # ← Correct
       - DATABASE_PASSWORD
   ```

2. Don't echo secrets in commands:
   ```yaml
   # Bad
   aliases:
     test: app exec "echo $DATABASE_PASSWORD"
   
   # Good
   aliases:
     test: app exec "rails test"
   ```
</Accordion>
